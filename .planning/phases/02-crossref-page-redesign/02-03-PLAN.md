## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Client → DOM | User interacts with uploaded file names rendered in the table |
| Client → API | HTTP requests carry file data and delete commands |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-02-07 | Tampering | File upload drop zone (drag-and-drop) | mitigate | All file uploads go through the same backend POST /upload endpoint which calls `validate_upload()` for extension/content check. Frontend accept attribute limits to .pdf/.csv/.ppt/.pptx/.doc/.docx but is not a security boundary — server validation is. |
| T-02-08 | Injection | File name rendering in table cell | mitigate | File names are rendered via React JSX (auto-escaped). No dangerouslySetInnerHTML or raw HTML injection. Backend also uses `sanitize_filename()` before storage. |
| T-02-09 | Spoofing | PillChip status display | accept | Status values come from the API response, not user input. The frontend merely renders what the backend provides. False status would require database tampering. |
| T-02-10 | Repudiation | File deletion via DELETE endpoint | mitigate | The backend has auth dependency (`require_auth`) and RLS policies scoping delete to `created_by`. The frontend calls the existing `api.crossref.delete()` which sends authenticated requests. |
</threat_model>
